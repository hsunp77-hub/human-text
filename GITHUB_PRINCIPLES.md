# 깃허브 협업 및 관리 핵심 원칙 (GitHub Core Principles)

이 원칙은 프로젝트의 안정성을 유지하고 원활한 협업을 위해 **커밋(Commit) 및 푸시(Push) 전에 반드시 확인**해야 합니다.

## 1. 메인 브랜치의 안정성 유지
- `main` (또는 `master`) 브랜치는 **언제나 정상적으로 동작하는 상태**여야 합니다.
- 불완전한 코드나 빌드 오류가 발생하는 코드는 절대로 메인 브랜치에 직접 커밋하거나 머지하지 않습니다.

## 2. 브랜치 전략 (Feature Branch)
- 새로운 기능을 추가하거나 버그를 수정할 때는 반드시 **새로운 브랜치**를 생성하여 작업합니다.
- 브랜치 이름은 작업 내용을 알 수 있도록 명확하게 정합니다. (예: `feature/home-redesign`, `fix/login-issue`)

## 3. 코드 리뷰 및 합의
- 작업이 완료되면 바로 머지하지 않고, **사용자와 함께 리뷰**를 진행합니다.
- AI 어시스턴트는 변경 사항을 명확히 설명하고, 사용자의 피드백을 반영합니다.

## 4. 메인 브랜치로의 머지
- 모든 기능이 정상적으로 동작하고, 사용자 및 AI 간의 **최종 합의가 이루어진 경우에만** 메인 브랜치로 머지합니다.
- 머지 후에는 메인 브랜치에서 다시 한번 정상 동작 여부를 확인합니다.

---
*이 원칙은 프로젝트의 품질과 지속 가능성을 위해 엄격히 준수합니다.*
